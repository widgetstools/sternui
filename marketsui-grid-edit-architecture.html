§ 01

Why one design, not three

Batch edit, paired tick, and undo all sit on top of the same primitive: a buffer of pending edits keyed by (rowId, colId) that the UI reads from, the user mutates, and the commit pipeline drains. Build them apart and they will fight over that buffer.

AG-Grid's native batch edit gives us startBatchEdit / commitBatchEdit / cancelBatchEdit, but it has no opinion on the two things we actually need on a trading desk: where edits come from (keyboard, fill handle, paired delta, paste from chat) and what survives a commit failure (server-acknowledged is not the same as commitBatchEdit returning).

The design here treats the pending-edit buffer as the single source of truth for in-flight changes. Paired tick is a writer against the buffer. Undo is a reader/writer that also tracks committed history. Batch edit is the commit boundary that flushes the buffer to the row data.

Decision

Single PendingEditBuffer owned by an edit-coordinator service. AG-Grid's batch state mirrors this buffer but is not authoritative.

Anti-pattern

Letting AG-Grid's cellEditRequest events drive three independent listeners. Race conditions on commit, no shared validation, undo loses paired context.

§ 02

The shared mental model

Three ideas the trader needs to feel without reading documentation:

  • Pending vs committed. A pending cell is yellow and dotted; the trader knows nothing has gone to the Street yet.
  • Linkage is visible. When two columns are paired, a chain glyph sits in the column header. Edits to one preview-shift the other. The trader can see the relationship before the edit lands.
  • Undo is for my edits. Sorts, filters, streaming ticks, and external feed updates are not in the undo stack. Only edits the trader made — including paired-tick fan-outs — can be reversed.
Design principle
Predictability beats power. Every interaction the trader makes either stages, commits, or reverses. A feature that does two of those things at once gets cut.

Boundaries that matter

L1 → L2 normalization

Every input device produces an Intent. A trader hitting Alt+↑ on a range selection produces a TickIntent with direction: 'up' and the column-aware tick size resolved later. A drag-fill produces a series of EditIntents. A paste produces EditIntents with a source: 'clipboard' tag so the validator can apply paste-specific rules. The intent layer is the place to tag origin.

L3 → L4 separation

The coordinator never writes to AG-Grid directly. It writes to the PendingEditBuffer, then asks AG-Grid to refresh affected cells. Two reasons: first, AG-Grid's batch API is one-way (you can't ask it for a structured diff cheaply); second, this layer is what the undo stack reads, and undo needs a stable, queryable representation.

L4 → L5 boundary

Commit is the only place a pending edit becomes durable. ViewServer publish and AG-Grid applyTransactionAsync both happen here. Critically, they are linked: an edit is not removed from the pending buffer until ViewServer acknowledges. We will come back to this in the commit pipeline section.

§ 04

State machine for an edit

Each entry in the PendingEditBuffer moves through the states below. The undo stack mirrors transitions for entries the trader caused.

State Meaning Visible to trader In undo
DRAFT Edit is in the buffer, not yet validated Yellow, dotted border No
VALID Passed validation, eligible for commit Yellow, solid border Yes (after commit)
REJECTED Validation failed; reason shown in tooltip Red diagonal hatch No
COMMITTING Sent to ViewServer, awaiting ack Pulsing yellow No (locked)
COMMITTED Server acknowledged; row data updated Brief green flash, then normal Yes (in committed history)
FAILED Server rejected; edit returns to buffer as REJECTED Red, with retry affordance No
ABANDONED Trader cancelled the batch Reverts to underlying value No

Transitions: DRAFT → VALID on validator pass; VALID → COMMITTING on commitBatchEdit; COMMITTING → COMMITTED on ViewServer ack; COMMITTING → FAILED on ViewServer NACK or timeout. VALID → ABANDONED on cancelBatchEdit.

Paired tick adjustment

The trader selects a range, hits Alt+↑ to shift bids by one tick. For paired columns (Bid + Ask, or Spread + Price), one keystroke produces two edits per affected row, both staged as pending.

Ctrl+Enter is overloaded — outside an active cell editor it commits the batch; inside one it applies the edited value to the range (AG-Grid native). The coordinator routes based on gridApi.getEditingCells().length. Document this clearly in trader onboarding.

The linkage configuration

Linkage is config-driven and lives in your hierarchical config system. A ColumnGroupLink describes how columns relate when one is edited. 

interface ColumnGroupLink {
  id: string;
  columns: string[];            // e.g. ['bidPrice', 'askPrice']
  strategy:                  // how to propagate
    | 'preserve_spread'      // move both by same delta
    | 'preserve_ratio'       // move proportionally
    | 'derive_from_anchor';  // recompute from formula
  tickSize: number | TickResolver;
  unit: 'bps' | 'price32nd' | 'pctPar' | 'absolute';
  enabled: boolean;          // trader can toggle the link
  formula?: (anchor, row) => Record<string, number>;
}

On a TickIntent, the LinkageResolver walks the selection, for each cell finds the linkage group that includes the edited column, applies the strategy, and emits one EditIntent per affected cell. All emitted edits share an intentId so undo treats them as a single unit.

Why the intentId matters

A trader hitting Alt+↑ once and then Ctrl+Z expects all 60 cells to revert, not just the last one written. The undo frame must reference the intentId, not individual cell positions, otherwise a 30-row tick takes 60 undos to reverse — and the second undo would partially break linkage invariants.

§ 06

Undo stack design

AG-Grid's built-in undo clears on sort, filter, or group. That is fatal for a blotter. Our undo is independent of AG-Grid's grid state and lives in L3.

Two-tier history

The undo subsystem tracks two stacks: a working stack (uncommitted) and a committed history. Different rules apply.

StackContentsReversible?
Working Pending edits not yet committed; undoable by buffer mutation only Yes, freely
Committed Edits acked by ViewServer; undo emits a new compensating edit Yes, but as a new edit (audit-trail preserved)

The trader experiences a single Ctrl+Z. Internally the stack chooses: if the top frame is in the working stack, revert the buffer. If the working stack is empty, generate a compensating edit from the top of the committed history and push it as a new pending edit. Compliance never sees data "disappear" — every committed change has a forward and reverse audit record.

Frame structure

interface UndoFrame {
  intentId: string;          // groups paired-tick fan-outs
  timestamp: number;
  trader: string;
  origin: 'keyboard' | 'fill' | 'paste'
        | 'pairedTick' | 'cellEditor';
  tier: 'working' | 'committed';
  edits: Array<{
    rowId: string;
    colId: string;
    before: CellValue;
    after: CellValue;
  }>;
}

What does NOT enter the stack

  • Streaming ticks from ViewServer (someone else's edit, not the trader's)
  • Sort, filter, group, column reorder (these are view operations, AG-Grid handles their own undo if at all)
  • Edits from a paired tick that violate validation (REJECTED never enters)
  • Server-side enrichment writes (e.g., FV recomputation on benchmark move)

Per-row undo

Right-click a row → "Undo my last 3 edits to this CUSIP". Implemented as a filter over the unified stack: walk frames newest-first, take only edits where rowId === target, generate compensating frame. Order matters — undo them in reverse-chronological order to preserve invariants.

Commit pipeline

Commit is the dangerous moment. Multiple edits go from local pending state to published axes on the Street, simultaneously, with the wrong-thing-published cost measured in basis points of P&L.

Why all-or-nothing on Phase 1

If 28 of 30 paired-tick edits validate but two fail (e.g., spread outside fat-finger threshold), do not commit the 28. Traders rely on the run being internally consistent — a half-published axe set is worse than no publish. Reject the whole batch, surface the two problem rows, let the trader fix or override.

Partial-ack reconciliation

ViewServer fans out to multiple venues; one venue may NACK while others ACK. The envelope tracks per-venue acknowledgment. On partial:

  • Cells that committed everywhere → COMMITTED, normal flow
  • Cells that committed somewhere → COMMITTED with venue-mismatch warning glyph; trader resolves
  • Cells that committed nowhere → FAILED, return to buffer with error

The reconciliation row is a real UI surface — a small status bar entry that lets the trader review per-venue state without leaving the grid.

§ 08

ViewServer interaction

Two flows touch ViewServer: outbound (commit → publish) and inbound (subscription updates → grid refresh). They must not collide.

Inbound during pending state

Trader has a pending edit on 912828YK0.bidPrice. ViewServer pushes a delta on the same cell from a different source (e.g., automated peer-pricing recomputation). What happens?

  • The underlying row data updates normally (committed value).
  • The pending overlay continues to show the trader's value — they retain control.
  • A small glyph appears on the cell: "underlying changed since you started editing".
  • On hover or expand, the trader sees both values and can choose to rebase or commit-as-is.

This is the same pattern Git uses for "your branch and origin have diverged". Traders understand it once and never get surprised again.

Subscription suspension during commit

For the duration of a commit envelope (typically < 200ms), the affected rows suspend inbound delta processing for the columns being committed. Otherwise a race between the COMMITTED ack and a stream tick can cause the cell to flicker between values. Suspension is column-scoped, not row-scoped — other columns continue to update.

Hooks into your existing SDK

The commit publisher is a thin wrapper around ViewServerGateway's publish path. The pending buffer integrates with useSubscription via a transform: the hook returns rows where overlay values mask underlying values when an entry exists in the buffer. 

const rows = useSubscription('fi.axes.live', {
  overlay: pendingBuffer,
  conflictPolicy: 'preserve-overlay-warn',
});

§ 10

Per-product configuration

The architecture is the same across all products. What changes is the configuration that flows through it: tick size, quote convention, fat-finger thresholds, linkage rules, lifecycle states, and the publish targets the commit pipeline talks to. This section is the reference matrix.

Each product card below specifies the values needed to instantiate ColumnGroupLink, TickResolver, and the validator thresholds for that asset class. These belong in your hierarchical config system as edit-policy artifacts that inherit from a desk-level default, with per-product overrides.

Rates · US Treasuries, Agencies, Strips

EXCHANGE-LIKE LIQUIDITY

Most liquid product on the desk. Tight spreads, fast tick, on-the-run vs off-the-run distinction matters. Quote convention is the legacy 32nds-with-plus that traders insist on keeping.

Quote unit
32nds of a point, with + denoting +½ tick (= 1/64). Quarter-tick suffixes 2 / + / 6 for ¼, ½, ¾ of a 32nd. Example: 99-03+ = 99 + 3.5/32 = 99.109375.
Min tick
1/256 for cash actives (eighth of 32nd); 1/128 for older issues; 1/32 for off-the-runs. Configurable per CUSIP via reference data.
Tick keystroke
Alt+↑/↓ = 1 minimum tick · Alt+Shift+↑/↓ = full 32nd · Ctrl+Alt+↑/↓ = full point
Linkage
{ bid, ask } with preserve_spread. Spread typically 1/256 for OTRs, 1/64 for OFRs. Yield column is derived (not directly editable) and updates on price tick.
Fat-finger
Reject edit if Δ > 8/32 (¼ point) from current mid. Hard reject > 1 full point.
Size convention
Notional in MM. Rounding to nearest 1MM on edit. Alt+Shift tick on size = 5MM increment.
Publish targets
BrokerTec, Tradeweb, Bloomberg ALLQ, Dealerweb. Most have streaming axe support — use the streaming publisher, not RFQ-by-RFQ.
Editor component
TreasuryPriceEditor — accepts 99-03+, 99-032, 99-03+2, decimal 99.109375; renders canonical 32nds form. Same parser handles paste from chat.
Gotcha When the trader pastes 99-03+ into a column configured as decimal, the parser must recognize the dash-and-plus signature and convert. Conversely, do not blindly format a decimal column as 32nds because the trader is "in rates" — yield columns are decimal even on a treasury blotter.

FX · Spot, Forwards, Swaps, NDFs

SUB-SECOND TICK

Fastest-moving product. Pip and pipette conventions; big-figure abbreviation in voice. Tiered pricing per client and per size band makes linkage non-trivial.

Quote unit
Pip = 4th decimal for most pairs (EUR/USD 1.1050, pip = 0.0001). For JPY pairs, pip = 2nd decimal (USD/JPY 154.02, pip = 0.01). Pipette = 1/10 pip (5th decimal, or 3rd for JPY).
Min tick
1 pipette in major pairs. Wider in exotics — config table per pair.
Tick keystroke
Alt+↑/↓ = 1 pipette · Alt+Shift+↑/↓ = 1 pip · Ctrl+Alt+↑/↓ = 10 pips (big-fig adjustment)
Linkage
Three-way: { bid, ask, mid } with preserve_spread. Mid is the anchor — editing mid moves both. Spread per pair from config (EUR/USD < 1 pip, exotics 5–50 pips).
Tiering
Per-client, per-size-band overlay. Editing the base price cascades to all tiers using derive_from_anchor: each tier has a config-driven add-on. Trader edits base; sees pending preview of all 12 tier prices before commit.
Fat-finger
Reject if Δ > 50 pips for majors, 200 pips for crosses. Hard cap: 2× normal day's range.
Big-fig handling
Voice paste "09-14" with no big-fig — editor pulls the big-fig from current market level and prompts trader to confirm if the inferred level is > 5 pips from mid.
Publish targets
EBS, Refinitiv FXall, internal SDP, Bloomberg FXGO. Streaming pricing typical. Subscription suspension during commit is critical here — pricing fans out 10+ times/sec.
Gotcha JPY pair detection cannot be string-matched on "JPY". Use the reference-data quotePrecision attribute, because some crosses (e.g., EUR/JPY) flip pip conventions and the parser will silently mis-format prices if you hardcode currency strings.

Corporates · IG, HY, EM

SPREAD-LED

The product where spread-to-benchmark is the trader's primary view, with price as derived. Linkage is bidirectional — edit either spread or price, the other follows.

Quote unit
Price as % of par to 1/8 or to 3 decimal places (99.875 or 99-7). Spread to benchmark in basis points (G-spread, T-spread, Z-spread, OAS depending on bond). Yield in % to 3 decimals.
Min tick
Price: 1/8 (legacy) or 0.001 (modern) — config per CUSIP. Spread: 0.5 bp. Yield: 0.001%.
Tick keystroke
Alt+↑/↓ on spread cell = 1 bp · Alt+Shift+↑/↓ = 5 bp · on price cell defaults to spread tick (re-prices through curve)
Linkage
Triangular: { price, spread, yield } with derive_from_anchor. Anchor depends on which cell trader edits. Re-pricing engine is server-side via ViewServer — pending state shows spinner glyph during round-trip (target: < 50ms).
Benchmark linkage
Spread is anchored to a benchmark CUSIP. When the benchmark ticks, all spreads on that benchmark recompute prices automatically — but those are system updates, not trader edits, and do not enter the undo stack.
Fat-finger
Spread Δ > 25 bp triggers warning, > 50 bp hard reject. IG and HY use different thresholds (HY can move 50 bp in a session, IG cannot).
Workup discount
Some trades go through workup protocol — initial level then size discovery. Edit pattern: stage initial level, post, then receive workup pings as separate inquiries. Each ping is a fresh row, not an edit to the original.
Publish targets
MarketAxess, Tradeweb, Bloomberg ALLQ, Trumid. RFQ-driven — most edits stage axes, few stream live.
Gotcha When a trader edits a spread, the price round-trips through the server. Two pending entries appear (spread = entered, price = computed-pending). Both share the same intentId so undo reverts together. The trader sees one yellow cell on spread and a yellow-spinner cell on price.

Credit · CDS, Index Tranches, CLO

PROTOCOL-HEAVY

CDS is a different beast — running spread vs upfront, ISDA conventions, points up-front for distressed names. Index products (CDX, iTraxx) trade like a single instrument but reference a basket.

Quote unit
Single-name CDS: running spread (bp) or points-up-front (% of notional) for stressed names. Index: spread or price (depending on series convention). CLO tranches: discount margin (DM) in bp over benchmark.
Min tick
CDS spread: 0.5 bp for liquid names, 1 bp for off-the-run. PUF: 0.125 (= 1/8 point). Index: 0.25 bp.
Tick keystroke
Same as corporates. Ctrl+Alt+↑/↓ on PUF cell jumps in 1-point increments.
Linkage
{ runningSpread, puf, cleanPrice } all derivable from each other given coupon convention and curve. Anchor configurable per protocol — distressed names pin to PUF, performing names pin to spread.
Convention switch
Same CUSIP can flip from spread-quoted to PUF-quoted as it stresses. The grid surfaces the active convention as a column-header badge; trader can toggle, which restages the column anchor.
Fat-finger
Spread Δ > 50 bp warning for performing IG, > 200 bp for HY. PUF Δ > 5 points = warning. The threshold table is per-rating-bucket.
Index roll
Twice yearly the on-the-run series rolls (e.g., CDX.NA.IG.41 → 42). Active-series flag in reference data; old series moves to off-the-run pricing rules. Edit policy follows the active series automatically.
Publish targets
MarkitWire, ICE Link for SEF-required products. Voice-still-dominant for some single-names.
Gotcha CDS notional is a face amount, but CDS P&L is based on PV01 (= price-value-of-1-bp). Size-related fat-finger thresholds must be expressed in PV01-equivalent, not raw notional, or a 100MM 5Y trade will look the same as a 100MM 30Y trade. Two very different risk transfers.

Munis · Tax-Exempt GO & Revenue, Taxable

YIELD-QUOTED · BID-WANTED

Yield-quoted serial bonds, dollar-quoted term bonds, and a bid-wanted dominant secondary market mean the edit workflow is fundamentally different. Less streaming, more inquiry-driven, MSRB G-18 compliance recordkeeping baked in.

Quote unit
Serial bonds: yield-to-worst (basis quote), e.g., 4.75 meaning 4.75% YTW. Dollar bonds (term, revenue): price as % of par in 1/8ths, like corporates. Same blotter must support both.
Min tick
Yield: 0.01% (1 bp YTW). Dollar bonds: 1/8 of a point.
Tick keystroke
On yield cell: Alt+↑ increases yield (which decreases price — this surprises new traders). Inverse-direction warning the first time per session, then suppressed. Alt+Shift+↑/↓ = 5 bp.
Linkage
For yield-quoted: { yield, equivPrice } with derive_from_anchor = yield. For dollar bonds: same linkage as corporates. Coupon, call schedule, and tax-exempt status drive the price/yield calc — the engine needs full reference data per CUSIP.
Bid-wanted protocol
Most muni inquiry is bid-wanted, not standing axes. The grid mode toggles: when in bid-wanted view, edits stage responses to specific inquiries, not standing prices. Commit publishes to the originating ATS (TMC, Bloomberg BWIC, MMD).
Fat-finger
Yield Δ > 25 bp warning, > 100 bp hard reject. Calibrated wider than IG corporates because muni curves move more on illiquid names.
MSRB G-18 capture
Every commit snapshots the comparables panel state (recent trades, dealer offers, model fair-value) into the audit log alongside the edit. Required for fair-pricing defense. The commit pipeline writes both.
Publish targets
ICE TMC, Bloomberg, Tradeweb Direct, MuniBrokers. Bid-wanteds also flow through the BWIC system with hard deadlines — commit pipeline must respect the BWIC cutoff time as a constraint.
Gotcha The same trader's blotter has yield-quoted serials in row 1 and dollar-quoted terms in row 2. The "Price" column header changes meaning per-row. Solution: use a virtual marketLevel column that displays per-row according to the bond's quote convention, with a small badge showing which convention applies. Do not have separate yield and price columns visible — clutter wins.

§ 11

Cross-product workflows

Several real workflows span products. The architecture handles them via the same intentId mechanism that groups paired-tick fan-outs — a single intent can write across products and is committed atomically.

Hedge-linked edits (rates ↔ corporates)

A corporate trader buys 5MM IG corporate, hedges with 4MM 5Y treasury short. On the blotter, the position rows are linked: editing the corporate's working size proposes a paired edit on the hedge size using the configured hedge ratio (DV01-matched, not notional-matched).

  • Hedge ratio is config per (corporate-CUSIP, treasury-CUSIP) pair, computed nightly from DV01s and refreshable on demand.
  • Edit on the corporate row stages a pending entry on the corporate row and on the linked treasury row, with shared intentId.
  • Trader sees both pending cells, with a small chain glyph between rows in the gutter. Either can be unlinked at edit time if the desk wants asymmetric exposure.
  • Commit publishes the corporate axe to MarketAxess and the treasury axe to BrokerTec — different venues, single envelope, all-or-nothing semantics.

Curve-shift bulk edit (rates)

Rate trader wants to mark the entire treasury curve up 1 bp in yield. Selects the curve column, applies a CurveShiftIntent. The intent expands to one EditIntent per CUSIP, each with the appropriate price delta computed from that bond's duration. Single undo frame reverses the entire curve shift.

Basis trade staging (rates ↔ futures)

Cash treasury vs treasury futures basis trade. Two rows linked with a conversion-factor-aware ratio. Editing the cash leg's size cascades to the futures leg's contract count via derive_from_anchor with a custom formula. Same intent, atomic commit, undo treats the pair as one.

Index roll (credit, FX-fwd)

Credit index roll twice a year, FX forward roll daily. Both involve cancelling axes on the old series/tenor and re-staging on the new. Modeled as a RollIntent that produces a paired (cancel, restage) edit on each affected CUSIP. The cancel and restage share an intent and commit as a single envelope; partial failure leaves nothing exposed.

Cross-product portfolio trade

Buy-side sends a 200-line portfolio trade across IG, HY, and EM. Sell-side prices the basket. The grid loads all 200 rows with a shared basketId. Tick adjustments apply within product (HY ticks in 1/8, IG in 0.5 bp); commit envelopes are partitioned by venue (MarketAxess for IG/HY, Trumid for HY portfolio protocol) but logically tied to one basketId.

The cross-product invariant: the trader sees one pending state for the basket; the commit pipeline handles per-venue envelopes; the audit log records one basket-level commit with venue-level acknowledgments rolled up.

Implementation note
Cross-product intents do not require new architecture — they are existing primitives composed via shared intentId and basketId. The work is in the linkage configuration (which rows are paired, with what ratio, expiring when) and in the venue-routing logic of the commit pipeline. Both are config-driven, not code paths.

Known pitfalls

Things to plan for now, not after the first production incident.

Pitfall · 1

Sorted/filtered grid + range selection. AG-Grid range selections are visual ranges, not row-id ranges. Resolve to row IDs at intent creation, not at commit, or the trader can sort mid-edit and apply the tick to the wrong rows.

Pitfall · 2

Undo across reconnect. WebSocket drops, trader reconnects. The undo stack must persist (sessionStorage or server-side per-trader) so a flaky network doesn't lose the morning's edits.

Pitfall · 3

Two traders, same desk, same row. Sell-side desks share books. Optimistic concurrency: stage edits include a baseVersion. Commit fails with a clear merge surface if a peer committed first.

Pitfall · 4

Performance with 600+ columns. Pending buffer keyed by (rowId, colId) Map. Don't iterate the buffer per render — emit cell-level subscriptions and re-render only affected cells via AG-Grid's refreshCells({ rowNodes, columns }).

Pitfall · 5

Tick size on derived columns. A trader ticks "yield" by 1bp; you compute price by re-pricing. The price column shows the derived pending value but does not enter the undo stack as a separate edit. Mark derived cells with a derivedFrom attribute and exclude them from undo frames.

Pitfall · 6

Read-only-edit mode. AG-Grid's read-only-edit emits cellEditRequest instead of writing. If you flip the grid into this mode, the entire flow above must route through that event. Pick one mode at coordinator init and stick to it.

Open questions for the next iteration

  • Should paired-tick respect a per-trader fat-finger override (sales-trader vs head-of-desk)?
  • Per-row undo across the committed tier — surface in UI as "show audit trail" or hide it?
  • Bulk paste of a 100-row spreadsheet from chat — stage as one frame or one per row?
  • Does the linkage config belong in the column config or in a separate edit-policy config artifact? Affects how it's promoted dev → UAT → prod.
